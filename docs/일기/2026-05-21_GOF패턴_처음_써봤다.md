## 계기

어제 9개 테이블 파이프라인을 다 만들었다.
근데 코드를 보니까 같은 코드가 9개 파일에 반복되고 있었다.

```python
# 이게 9개 파일에 거의 똑같이 있었음
async def ingest_store(self, session, year_quarter=None, region="seoul"):
    reader = MarketRadarReader(region)
    df = reader.load_store(year_quarter)
    df = df.rename(columns=KOREAN_TO_ENGLISH_COLUMN_MAP)
    df["region"] = region
    rows = df.to_dict(orient="records")
    count = await self.repository.bulk_insert(session, rows)
    return count
```

`load_store` 이 부분만 각자 다르고 나머지는 완전히 동일했다.

"이거 뭔가 잘못됐다"는 느낌은 왔는데, 어떻게 고쳐야 할지 몰랐다.

## Template Method 패턴

검색해보다가 이 상황에 딱 맞는 패턴을 찾았다.

**Template Method 패턴**: 알고리즘의 뼈대는 부모 클래스에 고정하고, 바뀌는 부분만 자식 클래스에서 구현하는 것.

이론으로 배울 때는 "흐름은 고정, 세부 구현은 서브클래스에" 라고 외웠는데
지금 상황이 딱 그거였다.

- **고정된 부분**: Reader로 읽기 → 컬럼명 변환 → region 추가 → DB 저장
- **바뀌는 부분**: 어떤 파일을 읽을지, 어떤 컬럼 맵을 쓸지

그래서 이렇게 만들었다:

```python
# 부모 클래스 — 흐름을 고정
class BaseIngestService(ABC):

    async def ingest(self, session, year_quarter=None, region="seoul"):
        reader = MarketRadarReader(region)
        df = self._load(reader, year_quarter)       # ← 자식이 구현
        df = df.rename(columns=self.column_map)     # ← 자식이 구현
        df["region"] = region
        rows = self._preprocess(df.to_dict(orient="records"))
        return await self.repository.bulk_insert(session, rows)

    @abstractmethod
    def _load(self, reader, year_quarter): ...      # 반드시 구현해야 함

    @property
    @abstractmethod
    def column_map(self): ...                       # 반드시 구현해야 함
```

```python
# 자식 클래스 — 달라지는 부분만 구현
class StoreService(BaseIngestService):

    @property
    def column_map(self):
        return KOREAN_TO_ENGLISH_COLUMN_MAP

    def _load(self, reader, year_quarter):
        return reader.load_store(year_quarter)

    def _create_repository(self):
        return StoreRepository()
```

처음에 `@abstractmethod`를 써야 한다는 걸 몰라서 에러가 났다.
ABC(Abstract Base Class)를 import 안 한 것도 실수했다.
이런 건 직접 써봐야 알게 되는 것 같다.

## Generic Repository 패턴

Repository도 똑같은 문제가 있었다.
`bulk_insert`와 `count`가 9개 파일에 그대로 복붙되어 있었다.

```python
# 9개 파일에 이게 똑같이 있었음
async def bulk_insert(self, session, rows):
    if not rows:
        return 0
    await session.execute(insert(StoreModel), rows)
    await session.commit()
    return len(rows)
```

여기서는 **Generic**을 처음 써봤다.

Generic이란 타입을 나중에 지정할 수 있게 해주는 것인데,
`BaseRepository[T]`에서 `T`가 나중에 `StoreModel`, `ApartmentModel` 등으로 바뀐다.

```python
class BaseRepository(ABC, Generic[T]):

    @property
    @abstractmethod
    def model_class(self) -> type[T]: ...   # T가 어떤 모델인지 알려주는 것

    async def bulk_insert(self, session, rows):
        await session.execute(insert(self.model_class), rows)  # T를 여기서 씀
        ...
```

```python
class StoreRepository(BaseRepository[StoreModel]):  # T = StoreModel

    @property
    def model_class(self):
        return StoreModel
```

이렇게 하면 `bulk_insert`를 한 번만 만들어도 모든 테이블에서 쓸 수 있다.
Generic이 이럴 때 쓰는 거구나 처음으로 체감했다.

## Mixin

모델에서는 Mixin을 적용했다.

9개 테이블 전부 `id`, `region`, `year_quarter`, `trdar_code` 같은 컬럼이 공통으로 있었다.
Mixin은 이런 공통 부분을 별도 클래스로 빼서 여러 클래스에 "섞어 넣는" 방법이다.

```python
class MarketStatMixin:
    id: Mapped[int] = mapped_column(primary_key=True)
    region: Mapped[str] = mapped_column(String(50), index=True)
    year_quarter: Mapped[int] = mapped_column(Integer, index=True)
    trdar_code: Mapped[int] = mapped_column(Integer, index=True)
    trdar_div_code: Mapped[str] = mapped_column(String)
    trdar_div_name: Mapped[str] = mapped_column(String)
    trdar_name: Mapped[str] = mapped_column(String)
```

```python
class StoreModel(MarketStatMixin, Base):
    # 이제 공통 컬럼 7개는 자동으로 들어옴
    # store만의 고유 컬럼만 적으면 됨
    service_code: Mapped[str] = ...
    store_count: Mapped[int] = ...
```

처음에 SQLAlchemy에서 Mixin이 제대로 작동할까 걱정했는데,
실제로 돌려보니까 테이블 정의가 이전과 완전히 동일하게 생성됐다.
DB를 다시 만들거나 데이터를 재적재할 필요가 없었다.

## 솔직히 헷갈렸던 것들

**1. Abstract vs Interface**

파이썬에는 interface가 없고 ABC로 비슷하게 구현한다는 걸 알고 있었는데,
`@abstractmethod`를 `@property`와 같이 쓸 때 순서가 헷갈렸다.

```python
# 이 순서가 맞다
@property
@abstractmethod
def column_map(self): ...

# 이건 틀림 (TypeError 발생)
@abstractmethod
@property
def column_map(self): ...
```

---

**2. `_load`에서 `year_quarter`를 무시하는 케이스**

Area 테이블은 분기 구분이 없어서 `year_quarter` 파라미터가 필요 없다.
근데 부모 클래스의 `ingest` 시그니처에 `year_quarter`가 있어서 처음엔 어떻게 처리해야 할지 몰랐다.

결국 그냥 무시하면 된다는 걸 알았다:

```python
class AreaService(BaseIngestService):
    def _load(self, reader, year_quarter):
        return reader.load_area()   # year_quarter 그냥 안 씀
```

파이썬은 파라미터를 받아도 안 써도 에러가 안 난다.

---

**3. `_nan_to_none`을 어디에 두냐**

float 컬럼에 NaN이 들어오면 DB가 에러를 낸다.
그래서 NaN을 None으로 바꾸는 처리가 필요한데,
이게 Apartment와 Consumption 두 개 서비스에만 필요했다.

처음엔 각 서비스 파일에 각각 정의했는데,
리팩토링하면서 `base_ingest_service.py`에 함수로 하나만 두고
필요한 서비스에서 `_preprocess`를 오버라이드해서 쓰는 방식으로 바꿨다.

```python
# 기본값: 아무것도 안 함
def _preprocess(self, rows):
    return rows

# ApartmentService, ConsumptionService만 오버라이드
def _preprocess(self, rows):
    return _nan_to_none(rows)
```

이게 Template Method 패턴의 "hook"이라는 개념이라는 걸 나중에 알았다.
훅: 자식이 원하면 오버라이드할 수 있지만, 안 해도 되는 메서드.

---

## 오늘 배운 것 정리

- **Template Method**: 흐름 고정, 세부 구현 위임. "공통 흐름이 반복되면" 적용
- **Generic Repository**: `Generic[T]`로 타입 파라미터화. "같은 로직인데 타입만 다를 때" 적용
- **Mixin**: 공통 속성 추출. "여러 클래스에 똑같은 필드가 반복될 때" 적용
- **abstractmethod**: 구현 강제. 자식 클래스가 반드시 구현해야 하는 메서드

이론으로 알고 있던 걸 실제로 써보니까 훨씬 와닿는다.
특히 "어떤 상황에서 쓰는지"가 이제 좀 보이는 것 같다.
