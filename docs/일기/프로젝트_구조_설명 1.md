## 전체 폴더 구조

```
backend/
├── apps/
│   ├── main.py                  ← FastAPI 진입점. 모든 라우터를 여기서 연결
│   ├── database.py              ← DB 연결 설정 (PostgreSQL)
│   │
│   ├── market/                  ← 상권 데이터 도메인 (핵심)
│   ├── auth/                    ← 회원가입 / 로그인
│   ├── chat/                    ← AI 채팅
│   ├── agent/                   ← AI 에이전트 (Navigator)
│   └── external/                ← 외부 API 클라이언트 (Gemini)
│
├── data/
│   └── raw/seoul/               ← 서울시 공공데이터 CSV 파일들
│
└── alembic/                     ← DB 마이그레이션 버전 관리
```

각 도메인(`market`, `auth`, `chat`)은 똑같은 내부 구조를 따른다.
이게 처음엔 왜 이렇게 폴더가 많나 싶었는데, 나중에 도메인이 늘어날수록 일관성이 있어서 편하다.

## 도메인 내부 구조 — market 기준

```
market/
└── app/
    ├── models/          ← DB 테이블 정의 (SQLAlchemy ORM)
    ├── repositories/    ← DB 쿼리 담당
    ├── services/        ← 비즈니스 로직
    ├── controllers/     ← FastAPI 라우터 (엔드포인트)
    ├── schemas/         ← 요청/응답 데이터 형식 (Pydantic)
    ├── readers/         ← CSV 파일 읽기
    └── utils/           ← 좌표 변환 등 유틸
```

## 각 레이어가 하는 일

### 1. models — DB 테이블 정의

파이썬 클래스로 DB 테이블을 표현한다.
실제 SQL을 직접 쓰는 게 아니라, 클래스 속성으로 컬럼을 정의하면 SQLAlchemy가 알아서 SQL로 변환해준다.

```python
class StoreModel(MarketStatMixin, Base):
    __tablename__ = "store"

    service_code: Mapped[str] = mapped_column(String, index=True)
    store_count: Mapped[int] = mapped_column(Integer)
    closure_rate: Mapped[int] = mapped_column(Integer)
    # ...
```

`MarketStatMixin`은 9개 테이블에 공통으로 있는 컬럼(`id`, `region`, `year_quarter`, `trdar_code` 등)을 묶어놓은 것이다.
이걸 상속받으면 공통 컬럼을 매번 적지 않아도 된다.

---

### 2. repositories — DB 쿼리

모델을 이용해서 실제 DB에 데이터를 넣거나 꺼내는 역할.
"어떻게 DB에 접근할지"만 다룬다.

```python
class StoreRepository(BaseRepository[StoreModel]):

    @property
    def model_class(self):
        return StoreModel

    async def find_by_trdar(self, session, trdar_code):
        # 특정 상권 코드로 데이터 조회
        ...
```

`BaseRepository`를 상속받으면 `bulk_insert`(여러 행을 한 번에 저장)와 `count`(총 개수 조회)는 자동으로 쓸 수 있다.
각 Repository는 그 테이블만의 특별한 조회 쿼리만 추가로 구현하면 된다.

---

### 3. services — 비즈니스 로직

"무슨 일을 할지"를 결정하는 곳.
Reader에서 데이터를 읽어오고, 필요한 가공을 한 뒤, Repository에 저장을 맡긴다.

```python
class StoreService(BaseIngestService):

    @property
    def column_map(self):
        return KOREAN_TO_ENGLISH_COLUMN_MAP   # 한국어 컬럼명 → 영어

    def _load(self, reader, year_quarter):
        return reader.load_store(year_quarter)  # CSV에서 읽기

    def _create_repository(self):
        return StoreRepository()
```

`BaseIngestService`를 상속받으면 실제 적재 흐름(`ingest` 메서드)은 부모 클래스가 처리해준다.
각 Service는 "어떤 파일을 읽을지", "어떤 컬럼 맵을 쓸지"만 알려주면 된다.

---

### 4. controllers — FastAPI 엔드포인트

HTTP 요청을 받아서 Service를 호출하고, 결과를 응답으로 돌려주는 곳.
"어떤 URL로 요청이 오면 무엇을 실행할지"를 정의한다.

```python
@router.post("/admin/ingest/store")
async def ingest_store(year_quarter: int | None = None, region: str = "seoul", db = Depends(get_db)):
    service = StoreService()
    count = await service.ingest(db, year_quarter, region)
    total = await service.repository.count(db)
    return {"inserted": count, "total": total}
```

## 데이터 흐름 — 공공데이터 적재 과정

```
CSV 파일
    ↓
MarketRadarReader.load_store()   ← readers: 파일을 읽어서 DataFrame 반환
    ↓
StoreService.ingest()            ← services: 컬럼명 변환, region 추가
    ↓
StoreRepository.bulk_insert()    ← repositories: DB에 저장
    ↓
PostgreSQL DB
```

## 데이터 흐름 — API 조회 과정

```

    ↓

    ↓

    ↓

    ↓
JSON 응답
```

## 아키텍처 계층 요약

| 계층 | 파일 위치 | 역할 | 의존 방향 |
|------|-----------|------|-----------|
| Controller | `controllers/` | HTTP 요청/응답 | Service 호출 |
| Service | `services/` | 비즈니스 로직 | Repository + Reader 호출 |
| Repository | `repositories/` | DB 쿼리 | Model 사용 |
| Model | `models/` | DB 스키마 정의 | DB 직접 매핑 |
| Reader | `readers/` | 외부 파일 읽기 | pandas |
| Schema | `schemas/` | 입출력 형식 | Pydantic |

각 계층은 바로 아래 계층만 호출한다.
Controller가 Repository를 직접 부르거나, Service가 Controller를 부르는 일은 없다.
이렇게 해야 나중에 수정할 때 영향 범위를 예측할 수 있다.
