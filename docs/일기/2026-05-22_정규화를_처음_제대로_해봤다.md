# 2026.05.22 — 정규화를 처음 제대로 해봤다

---

오늘은 코드를 한 줄도 안 짰다.
ERD를 보면서 "이 테이블이 왜 이렇게 생겼지?"를 처음으로 진지하게 생각해봤다.

---

## 계기

MARKET_ERD.md를 다시 보다가 이상한 게 눈에 밟혔다.

```
MarketStatMixin (8개 통계 테이블이 공통으로 가지는 컬럼)
  - trdar_code     ← 상권 코드
  - trdar_name     ← 상권명
  - trdar_div_code ← 상권 구분 코드
  - trdar_div_name ← 상권 구분명
```

그런데 AREA 테이블을 보니까:

```
AREA
  - trdar_code     ← 동일
  - trdar_name     ← 동일
  - trdar_div_code ← 동일
  - trdar_div_name ← 동일
```

완전히 똑같은 컬럼이 AREA에도 있었다.
8개 테이블 전부 상권명을 들고 다니고 있었던 것이다.
데이터가 수십만 행이니까 상권명이 수십만 번 반복 저장되고 있었다.

"이거 그냥 AREA 조인하면 되는 거 아니야?"

---

## 정규화가 뭔지 찾아봤다


**1NF (제1정규형)**
- 모든 컬럼이 원자값이어야 한다 (배열이나 중첩 값 없이)
- 반복되는 컬럼 그룹이 없어야 한다
- PK가 있어야 한다

현재 ERD는 1NF는 충족하고 있었다.
`mon_sales_amount`, `tue_sales_amount` 같은 컬럼들이 반복 그룹처럼 보이지만,
각각이 원자값이고 행이 중복되지 않으니까 1NF는 OK였다.

**2NF (제2정규형)**
- 1NF를 만족하면서
- 복합 PK의 일부에만 종속된 컬럼이 없어야 한다 (부분 함수 종속 제거)

이게 오늘의 핵심이었다.

---

## 2NF 위반을 세 군데 찾았다

### 1. MarketStatMixin의 trdar_name, trdar_div_code, trdar_div_name

통계 테이블의 복합 unique key는 `(trdar_code, region, year_quarter, ...)` 인데,
`trdar_name`은 `trdar_code` 하나에만 종속된다.

```
(trdar_code, region, year_quarter) → 전체 PK
 trdar_code                        → trdar_name  ← 부분 종속!
```

trdar_code만 알면 trdar_name은 항상 고정이다.
year_quarter가 달라도, region이 달라도 상권명은 바뀌지 않는다.

이미 AREA 테이블에 있으니까 8개 테이블에서 이 세 컬럼을 전부 빼면 된다.
조회할 때 AREA 조인하면 그만이다.

### 2. STORE, ESTIMATED_SALES의 service_name

```
(year_quarter, trdar_code, service_code, region) → 복합 PK
 service_code                                     → service_name  ← 부분 종속!
```

`CS101`이라는 service_code가 있으면 service_name은 항상 `한식음식점`이다.
분기가 달라져도, 상권이 달라져도 업종명은 안 바뀐다.

SERVICE(service_code PK, service_name) 테이블로 분리하면 된다.

### 3. COMMERCIAL_CHANGE의 change_indicator_name

```
change_indicator      → change_indicator_name  ← 부분 종속
```

이것도 마찬가지로 CHANGE_INDICATOR 테이블로 분리 대상이다.

---

## 솔직히 처음엔 "굳이 왜?" 싶었다

trdar_name을 8개 테이블에서 빼면 조회할 때마다 AREA를 조인해야 한다.
그게 오히려 불편하지 않나?

조금 더 생각해보니까 이해됐다.

만약 상권명이 바뀐다면 (실제로 행정구역명은 바뀌기도 한다),
정규화 안 된 구조에서는 8개 테이블을 전부 업데이트해야 한다.
정규화된 구조에서는 AREA 한 행만 바꾸면 끝이다.

그리고 저장 공간 문제도 있다.
trdar_name이 최대 30자라고 하면,
8개 테이블 × 85,000행 × 30바이트 = 약 20MB가 중복 저장되고 있는 셈이다.

---

## AREA 테이블은 3NF 위반이 남아있다

오늘 작업 범위는 2NF까지였는데,
AREA 테이블을 보다 보니 3NF 위반이 눈에 보였다.

```
AREA
  - district_code
  - district_name  ← district_code → district_name (이행적 종속)
  - adm_dong_code
  - adm_dong_name  ← adm_dong_code → adm_dong_name (이행적 종속)
```

이게 3NF 위반이다.
PK를 통하지 않고 다른 컬럼을 거쳐서 종속되는 경우다.

DISTRICT(district_code PK, district_name), ADM_DONG(adm_dong_code PK, adm_dong_name) 으로 분리할 수 있지만,
이건 실익이 크지 않다고 판단해서 일단 남겨뒀다.
자치구명이나 행정동명이 바뀌는 일이 얼마나 자주 있겠어.

---

## 오늘 배운 것

- **2NF 위반의 감**: 복합 PK인데 어떤 컬럼이 "PK 전체가 아니라 일부에만" 따라오면 의심한다
- **정규화의 목적**: 저장 효율보다 **수정 이상(update anomaly) 방지**가 더 중요한 이유다
- **3NF 판단**: 이론적으로 위반이어도 실무에서는 변경 빈도가 낮으면 그냥 두기도 한다

정규화는 "무조건 다 분리"가 아니라 "어디서 문제가 생기는지"를 보는 것이구나 싶었다.
